The fundamental theorem of software engineering is that there are no perfect solutions, only trade-offs. 

When designing ClotLang, every feature added was a conscious choice to sacrifice something else, be it raw performance, binary size, or cognitive simplicity. This page documents those deliberate decisions so you can understand the boundaries of the language and use it effectively.

## 1. Dynamic typing by default vs. Peak performance

**The Decision:** ClotLang variables are dynamic by default, and the default integer type (`int`) is a dynamic BigInt.

### The Mitigation: Progressive Strictness
To prevent this trade-off from making the language unusable for serious systems, we introduced **Two-Phase Typing** and explicit numerical types (`long`, `byte`, `double`). You can start dynamic, and enforce strictness later.

    ```clot
    datos = [1, "dos", true];
    ```

    ```clot
    func process_user(id: long):
        // Signature guarantees 'id' is exactly a 64-bit signed int
        return db.fetch(id);
    endfunc
    ```

## 2. Multiparadigm flexibility vs. Cognitive overhead

**The Decision:** ClotLang supports imperative procedural programming, an Object-Oriented Programming (OOP) MVP (`class`, `interface`, `extends`), and lightweight functional concepts (first-class functions).

<Cards>
  <Card 
    icon={<BrainCircuitIcon />} 
    title="The Benefit: Expressiveness" 
    description="You can model a complex domain using OOP interfaces, but write a quick data-processing script using functional maps and imperative for-loops." 
  />
  <Card 
    icon={<ShieldAlertIcon />} 
    title="The Cost: The 'Right Way' Dilemma" 
    description="Having multiple ways to solve a problem increases the mental burden on the developer. Teams must agree on conventions." 
  />
</Cards>

Unlike languages like Python (which strives for "one obvious way to do it") or Go (which heavily restricts features to keep the language small), ClotLang gives you a wider toolkit. The trade-off is that **you** are responsible for choosing the right tool. 

<Callout type="warn" title="Architectural responsibility">
  Because ClotLang doesn't force a single paradigm, a poorly structured ClotLang project can become a messy hybrid of deep class hierarchies mixed with mutated global state. Read the [Effective ClotLang](/docs/idiomatic) guide to establish team conventions.
</Callout>

## 3. AOT Compilation vs. Dynamic runtime features

**The Decision:** Implementing an Ahead-of-Time (AOT) LLVM compiler alongside the dynamic interpreter.

One of the hardest trade-offs in modern language design is bringing dynamic features (like `in` operators, `for-each` loops over heterogeneous objects, or `defer` blocks) into an AOT-compiled binary. LLVM IR is statically typed and low-level; it does not natively understand a ClotLang `object` or dynamic dispatch.

<div className="relative border-l-4 border-primary pl-6 py-2 my-8">
  <h3 className="text-lg font-bold mt-0">The Runtime Bridge Solution</h3>
  <p className="text-muted-foreground"> Instead of crippling the compiler by banning dynamic features, or crippling the interpreter by forcing everything to be statically typed, we built the **Runtime Bridge**. </p>
  <p className="text-muted-foreground mt-2"> When you compile with <code>--mode compile</code>, native math and basic logic are lowered directly to optimized LLVM IR. When the compiler encounters a complex dynamic feature (like a <code>switch</code> on a dynamic type), it injects a call to the Runtime Bridge. </p>
</div>

### The cost of the bridge
* **Binary Size:** A compiled ClotLang executable (`.exe`) is larger than an equivalent C binary because it must bundle the bridge.
* **Performance Drops:** Crossing the boundary from optimized LLVM IR into the C++ runtime bridge incurs a slight performance penalty compared to pure, raw LLVM execution.

## 4. Safety boilerplate vs. "Cowboy" coding

**The Decision:** Disallowing empty variable declarations and mandating explicit nulls.

In C or Java, you can declare `int x;` and assign it later. In JavaScript, an uninitialized variable is `undefined`, which silently propagates and causes chaos later in the application.

In ClotLang, **empty declarations are a syntax error.**
