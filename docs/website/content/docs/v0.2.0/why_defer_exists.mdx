In any programming language, acquiring a resource, whether it is opening a file descriptor, acquiring a mutex lock, or opening a network connection comes with a strict obligation: **you must release it when you are done.**

Failing to do so leads to resource leaks, deadlocks, and system crashes. However, ensuring that cleanup code runs reliably is notoriously difficult when a function has multiple `return` statements or when an exception (`throw`) interrupts the normal control flow.

This is the exact problem the `defer` keyword was designed to solve.

---

## The problem: Control flow complexity

Without a dedicated mechanism for deferred execution, developers usually have to manually duplicate cleanup code before every possible exit point of a function. 

<Tabs items={['The Messy Way (Without defer)', 'The ClotLang Way (With defer)']}>
  <Tab value="The Messy Way (Without defer)">
    ```clot
    func process_data(path: string):
        file = open_stream(path);
        
        if !file:
            return null;
        endif

        try:
            data = read_chunk(file);
            if data == "EOF":
                close_stream(file); // Cleanup 1
                return false;
            endif
            
            process(data);
            
            close_stream(file); // Cleanup 2
            return true;
            
        catch(err):
            close_stream(file); // Cleanup 3
            throw(err);
        endtry
    endfunc
    ```
    *Notice how `close_stream(file)` is repeated three times. If you add another early return, you might forget to close the file.*
  </Tab>
  <Tab value="The ClotLang Way (With defer)">
    ```clot
    func process_data(path: string):
        file = open_stream(path);
        if !file: return null; endif
        
        // Scheduled to run when the block exits, no matter how.
        defer close_stream(file); 
        
        data = read_chunk(file);
        if data == "EOF":
            return false;
        endif
        
        process(data);
        return true;
    endfunc
    ```
    *The cleanup is declared exactly where the resource is allocated. It is impossible to forget, and the code is drastically shorter.*
  </Tab>
</Tabs>

## How `defer` works under the hood

When the ClotLang compiler/interpreter encounters a `defer` statement, it does not execute the action immediately. Instead, it pushes the statement onto a hidden stack tied to the current execution block.

When the block scope ends whether by reaching the end of the block naturally, hitting a `return`, or throwing an exception the runtime pops and executes all deferred statements in **Last-In, First-Out (LIFO)** order.
