In language design, handling collections (like lists and maps) usually forces a stark choice. 

Dynamic languages (like Python or JavaScript) give you heterogeneous "buckets" where you can throw anything into an array, making prototyping incredibly fast but large-scale refactoring a nightmare. Static languages (like C++ or Java) force you to declare exactly what goes inside the array up front, providing massive safety but slowing down early development.

ClotLang refuses to make you choose. Instead, it introduces **Two-Phase Typing**, a progressive approach to data structures that grows with your codebase.

---

## The Core Concept

Two-Phase Typing separates the validation of the container itself from the validation of the data inside it.

## Why did we design it this way?

We built Two-Phase Typing because the lifecycle of a variable changes as a project matures.

### 1. The Prototyping Reality
When you are first integrating with a third-party REST API, you usually receive a messy JSON payload. At this stage, forcing the developer to define a rigid `list<map<string, int>>` before they even know what the data looks like is a massive friction point. 

Phase 1 (`list`, `object`, `map`) acts as a shock absorber. It lets you ingest, inspect, and manipulate diverse data quickly.

### 2. The Production Reality
Once the prototype works, you need to turn it into a reliable library or service. When you pass a list of user IDs to a critical database deletion function, you absolutely cannot risk a `string` slipping into that list. 

Phase 2 (`list<int>`, `object<User>`) acts as the structural steel. By adding generics to your hints, ClotLang's runtime validator steps in and provides absolute guarantees.

## When does this distinction matter in practice?

The true power of Two-Phase Typing becomes apparent in function signatures. It allows you to design APIs that are exactly as strict as they need to be.

<Tabs items={['Flexible Inputs (Phase 1)', 'Strict Boundaries (Phase 2)']}>
  <Tab value="Flexible Inputs (Phase 1)">
    ```clot
    // This function accepts ANY list. 
    // It doesn't care if it's a list of ints, strings, or objects.
    func print_length(items: list):
        println("The list has " + len(items) + " items.");
    endfunc

    print_length([1, 2, 3]);       // Valid
    print_length(["a", true, 9]);  // Valid
    ```
    *Use Phase 1 when you are writing generic utility functions that only care about the container's behavior (like iteration or counting), not the data itself.*
  </Tab>
  <Tab value="Strict Boundaries (Phase 2)">
    ```clot
    // This function demands a list of integers. 
    // The runtime will actively reject anything else.
    func calculate_sum(numbers: list<int>):
        sum = 0;
        for n in numbers:
            sum += n;
        endfor
        return sum;
    endfunc

    calculate_sum([1, 2, 3]);       // Valid
    calculate_sum([1, "two", 3]);   // Runtime TypeError
    ```
    *Use Phase 2 when your logic depends on the mathematical or structural properties of the elements inside the collection.*
  </Tab>
</Tabs>

## The Runtime Difference (No Type Erasure)

It is crucial to understand that ClotLang is not TypeScript. In TypeScript, generic types (`Array<number>`) exist only in the editor; they are erased during compilation to JavaScript.

In ClotLang, **Phase 2 typing is enforced at runtime.** If you declare a `list<int>` and attempt to `.append("hello")` or pass a `list<string>` into a `list<int>` parameter, the ClotLang runtime validator will intervene and throw a `TypeError`. This ensures that your Phase 2 guarantees hold true even when interacting with highly dynamic parts of your codebase or when executing via the Interpreter.

<Callout type="tip" title="Best Practice">
  Start your scripts with Phase 1 typing. As the script grows into a module or application, selectively upgrade the boundaries (function arguments and return types) to Phase 2 typing. This gives you the speed of Python with the boundary safety of C#.
</Callout>
